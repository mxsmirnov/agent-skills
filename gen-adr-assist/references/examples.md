# Некоторые примеры
Представленные в этом файле примеры носят вспомогательный характер

## Пример короткого ADR

```md
# Выбрать PostgreSQL как основную транзакционную СУБД

## Контекст (Context)
- Новой биллинговой системе нужна транзакционная СУБД.
- Команда уже поддерживает PostgreSQL.
- Нужны ACID-транзакции и стандартные средства резервного копирования.

## Драйверы решения (Decision Drivers)
- reliability
- team skills
- cost predictability

## Варианты (Options)
### 1. MySQL
- Плюсы (Pros): известная СУБД, команда умеет сопровождать
- Минусы (Cons): ограниченное применение для промышленных решений

### 2. Вендорская managed NoSQL DB
- Плюсы (Pros): меньше операционной нагрузки
- Минусы (Cons): слабее fit для транзакционного сценария, выше lock-in

## Решение (Decision)
Использовать PostgreSQL как основную транзакционную СУБД сервиса billing.

## Последствия (Consequences)
- Положительные: быстрый старт на знакомом стеке
- Отрицательные: команда отвечает за схему, миграции и HA-дизайн

## Допущения и открытые вопросы (Assumptions / Open Questions)
- Нагрузочный профиль укладывается в вертикальное масштабирование на первом этапе
- Нужно уточнить целевой RPO/RTO
```

## Пример реакции на `hold`

Если пользователь предлагает технологию, для которой `ring = hold`, модель должна ответить по сути так:
- решение в текущем виде отклоняется;
- причина: технология находится в `hold` в `tech-radar.json` навыка;
- далее предложить допустимые альтернативы и зафиксировать исходный вариант как rejected option.
